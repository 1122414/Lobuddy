# Lobuddy Repair Plan - 2026-04-25 Round 1

## 背景

当前 Lobuddy 已经完成了基础桌宠 UI、聊天窗口、设置窗口、历史窗口和任务状态窗口，但从最新截图和实际使用反馈看，仍然存在一批会严重影响基础体验的问题。

本轮修复目标不是重构整个项目，而是围绕 **设置同步、历史会话、任务窗口遮挡、消息渲染、窗口交互** 这几条主链路做稳定性修复。

请优先做小范围、可验证修改，不要引入大型新框架，不要重写现有 UI，不要改变当前整体橙色 / 奶油色风格。

---

## Bug 1：打开设置窗口后，其余 UI 无法点击，导致程序像“卡死”

### 现象描述

从截图可以看到，`Lobuddy Settings` 设置窗口打开后，背后的聊天窗口和桌宠控制台无法继续交互。用户在修改设置时，整个程序其余区域都处于不可点击状态。

这会造成几个问题：

1. 用户想一边看聊天窗口里的错误信息，一边调整设置，但无法操作。
2. 设置窗口如果被桌宠、聊天窗口或其他浮窗挡住，用户会觉得程序不可用。
3. 设置窗口看起来像普通工具窗口，但实际表现像阻塞式 modal dialog。
4. 如果设置窗口没有正确关闭，可能导致主界面长期不能点击。

### 复现步骤

1. 启动 Lobuddy。
2. 打开聊天窗口。
3. 点击齿轮按钮打开 `Lobuddy Settings`。
4. 不关闭设置窗口，尝试点击聊天输入框、发送按钮、历史按钮、桌宠控制台按钮。
5. 发现其余 UI 无法点击。

### 预期行为

设置窗口应该是 **非阻塞窗口**。

用户打开设置窗口后：

- 聊天窗口仍然可以滚动和查看。
- 桌宠控制台仍然可以点击。
- 用户可以切换窗口焦点。
- 设置窗口可以置顶显示，但不能阻塞整个应用。

### 重点排查方向

请检查设置窗口打开方式：

- 是否使用了 `dialog.exec()` / `exec_()`。
- 是否设置了 `ApplicationModal` 或 `WindowModal`。
- 是否应改为 `dialog.show()`。
- 是否错误设置了 `setModal(True)`。
- 是否因为 parent 关系导致主窗口被阻塞。

### 修复要求

1. 设置窗口必须改为非阻塞显示。
2. 不要用 `exec()` 打开设置窗口。
3. 推荐使用单例式设置窗口：如果已经打开，再次点击齿轮只把已有窗口 raise 到前台。
4. 设置窗口关闭后引用要正确释放或复用，不能产生多个重复设置窗口。
5. 设置窗口可以保持 `Qt.Tool` 或普通窗口样式，但不要阻塞主界面。

### 验收标准

- 打开设置窗口后，聊天窗口仍可滚动。
- 打开设置窗口后，桌宠控制台仍可点击。
- 再次点击齿轮不会无限创建多个设置窗口。
- 关闭设置窗口后程序交互正常。

## Bug 2：刚开始会话时 Task 任务窗覆盖聊天内容和输入区域，体验很差

### 现象描述

截图中可以看到，任务状态窗口在会话刚开始时直接浮在聊天窗口上方，遮挡了：

- 顶部标题栏区域
- 聊天消息内容
- 输入框附近区域
- 当前用户正在看的对话内容

而且任务窗口本身还有多层状态条，例如：

- `Running...`
- `Processing your request...`
- 多条空白状态条
- 下方按钮区域

整个窗口视觉上非常压迫，像错误弹窗覆盖在主流程上。

### 复现步骤

1. 启动 Lobuddy。
2. 在聊天框输入一句话，例如“你好，我是谁”。
3. 点击 Send。
4. 观察任务状态窗口出现的位置。
5. 发现它覆盖在聊天主窗口上，影响阅读和继续输入。

### 预期行为

任务状态不应该遮挡主对话区域。

更合理的方式是：

- 任务进行中只显示一个轻量状态提示。
- 可以放在聊天窗口顶部右侧、底部状态栏、或桌宠旁边。
- 不应该默认弹出大面积浮窗覆盖聊天内容。
- 任务完成后再显示轻量 toast 或在消息气泡中展示结果。

### 修复要求

本轮不要重做整套任务系统，只做体验修复：

1. 任务窗口默认不要覆盖聊天内容中心区域。
2. 任务进行中建议改为以下任一方式：
   - 聊天输入框上方显示一行小状态：`Working...`。
   - 聊天窗口右下角显示 toast。
   - 桌宠旁边显示小气泡状态。
3. 如果保留 Task 窗口，则必须：
   - 默认出现在聊天窗口外侧或右上角非关键区域。
   - 面积明显缩小。
   - 不遮挡输入框。
   - 不遮挡最新消息。
4. `Running...` 和 `Processing your request...` 不要重复堆叠成多条大横条。
5. 空白状态条不能出现。
6. 任务失败时可以显示错误详情，但默认折叠，不要大面积展开。

### 验收标准

- 发送消息后，用户仍然可以看到最新聊天内容。
- 发送消息后，输入框不被任务窗口遮挡。
- 任务状态不会出现多条空白进度条。
- 任务完成或失败后不会把旧的大弹窗留在屏幕中央。

## Bug 3：设置页面修改 setting 不会同步到 `.env`，`.env` 中的 setting 也不会同步到设置窗口

### 现象描述

当前设置窗口里能看到以下字段：

- `Pet Name`
- `LLM API Key`
- `LLM Base URL`
- `LLM Model`
- `Task Timeout`
- `Enable Shell Tool`

但是实际表现是：

1. 在设置窗口里改了配置，点击 Save 后，并没有同步写入 `.env`。
2. `.env` 文件中已经存在的配置，也不会正确同步显示到设置窗口。
3. 用户现在想改配置，仍然只能手动修改 `.env` 文件。
4. 设置窗口看起来可用，但实际上没有成为真正配置入口。

截图中设置窗口仍然显示：

```txt
LLM Base URL: https://dashscope.aliyuncs.com/compatible-mode/v1
LLM Model: kimi-k2.5
```

同时聊天中仍然出现 API Key 未提供的错误，说明 UI、`.env`、运行时 LLM client 三者没有统一。

### 复现步骤

1. 修改 `.env` 中的 LLM 配置。
2. 启动 Lobuddy。
3. 打开设置窗口。
4. 检查设置窗口是否显示 `.env` 中的真实值。
5. 在设置窗口中修改配置并点击 Save。
6. 打开 `.env` 文件检查是否被更新。
7. 发送消息检查运行时是否使用了新配置。

### 当前问题判断

这很可能不是单个 UI 问题，而是配置链路没有统一：

```txt
.env 文件
   ↓
配置加载模块
   ↓
Settings UI
   ↓
保存回 .env
   ↓
运行时 Config 对象
   ↓
LLM Client
```

现在很可能每一层都在各读各的，或者设置窗口只是显示默认值，没有真正读写配置源。

### 预期行为

配置应该有唯一可信来源。

推荐行为：

1. 程序启动时从 `.env` 读取配置。
2. 设置窗口打开时从统一 ConfigManager 读取当前配置。
3. 用户点击 Save 后：
   - 写入 `.env`。
   - 更新内存中的配置对象。
   - 通知 LLM client 使用新配置。
4. 再次打开设置窗口时，显示刚刚保存的值。
5. 重启应用后，配置仍然存在。

### 建议统一字段

请检查并统一环境变量名，不要在不同文件里混用多个名字。

建议统一为：

```env
LOBUDDY_PET_NAME=MyBuddy
LLM_API_KEY=
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL=kimi-k2.5
LLM_PROVIDER=dashscope
TASK_TIMEOUT_SECONDS=120
ENABLE_SHELL_TOOL=true
```

如果项目中已经有既有命名，也可以保留兼容，但必须明确优先级。

### API Key 特殊要求

API Key 输入框是密码模式，所以特别容易出现保存 bug。

必须满足：

1. 设置窗口打开时，如果已有 API Key，可以显示为密码点，但内部必须保存真实值。
2. 用户没有修改 API Key 时，点击 Save 不能把 API Key 覆盖为空或覆盖成 `••••••`。
3. 用户点击 Show 后可以查看真实值。
4. 用户修改 API Key 后，保存真实输入值。
5. 日志中禁止打印完整 API Key。

### 修复要求

1. 新增或修复统一配置管理模块，例如 `ConfigManager`。
2. 所有配置读取都走同一个入口。
3. 所有配置保存都走同一个入口。
4. 设置窗口打开时必须从当前配置初始化字段。
5. Save 后必须写回 `.env`。
6. Save 后必须刷新运行时配置。
7. LLM 请求前必须读取最新配置。
8. 如果 API Key 缺失，不要发请求，直接在 UI 中提示用户配置。

### 验收标准

- 修改设置窗口中的 `Pet Name`，保存后 `.env` 中同步变化。
- 修改 `LLM_MODEL`，保存后 `.env` 中同步变化。
- 修改 `Task Timeout`，保存后 `.env` 中同步变化。
- 重启程序后设置窗口显示 `.env` 里的值。
- 设置窗口保存后，不需要重启也能使用新 LLM 配置。
- API Key 不会被保存成空字符串、星号或圆点。

## Bug 4：历史对话明明有 messages，但 History 窗口仍显示 `0 messages`

### 现象描述

历史窗口现在已经不再是全黑弹窗，能显示历史列表，例如：

- `你好，我是谁`
- `你好`
- `New Chat`
- `打开edge浏览器`
- `帮我打开谷歌浏览器`

但是每条历史下面都显示：

```txt
0 messages
```

这明显不对，因为这些会话标题来自真实聊天内容，说明至少保存过部分会话信息。但消息计数却永远为 0。

### 复现步骤

1. 启动 Lobuddy。
2. 发送一条消息，例如“你好，我是谁”。
3. 等待 AI 回复。
4. 打开 History。
5. 找到刚刚的会话。
6. 发现该会话显示 `0 messages`。

### 可能原因

请重点排查：

1. 会话 metadata 中保存了标题，但 messages 数组没有保存。
2. 历史窗口读取的是 `message_count` 字段，但保存时没有更新该字段。
3. 实际消息保存在别的字段，例如 `chat_history` / `conversation`，但 UI 只读取 `messages`。
4. 历史列表只读取了会话索引文件，没有读取具体 session 文件。
5. 新建会话时创建了空 session，标题更新了，但消息没有 append。
6. 用户消息和助手消息只显示在 UI，没有写入持久化文件。

### 预期行为

History 中每条会话应显示真实消息数量。

例如：

```txt
你好，我是谁
4 messages
16:10
```

如果一个会话中用户问了 2 个问题，助手回答了 2 次，则应该至少显示 4 messages。

### 修复要求

1. 明确会话数据结构。
2. 每个 session 文件中必须有 `messages` 数组。
3. 用户发送消息后立即 append 一条 user message。
4. 助手回复完成后 append 一条 assistant message。
5. 历史窗口读取 `len(messages)` 作为消息数量。
6. 如果存在旧历史数据格式，要做兼容迁移或 fallback 读取。
7. 如果消息数量读取失败，不要显示错误的 0，应该记录 warning。

### 建议数据结构

```json
{
  "session_id": "session_xxx",
  "title": "你好，我是谁",
  "created_at": "2026-04-28T16:10:00",
  "updated_at": "2026-04-28T16:12:00",
  "messages": [
    {
      "role": "user",
      "content": "你好，我是谁",
      "timestamp": "2026-04-28T16:10:00",
      "metadata": {}
    },
    {
      "role": "assistant",
      "content": "你好！根据我的记忆，你是小明...",
      "timestamp": "2026-04-28T16:10:20",
      "metadata": {}
    }
  ]
}
```

### 验收标准

- 新会话发送 1 问 1 答后，History 显示 `2 messages`。
- 新会话发送 2 问 2 答后，History 显示 `4 messages`。
- 重启程序后消息数量仍正确。
- 点击历史会话后能恢复对应 messages。

## Bug 5：聊天气泡中有时回答不完整，UI 显示省略号，但 AI 实际已经返回完整回答

### 现象描述

截图中用户问：

```txt
我刚刚都问了什么问题
```

AI 回复中第二条历史被显示为：

```txt
2. "你搞错了，里面不都是陈金保开头的吗"
   - 你纠正了我，我重新确认后确实...
```

这里出现了省略号 `...`，但用户判断 AI 实际上已经给出了完整回答，只是 UI 中没有完整展示。

这说明问题很可能出在消息气泡渲染层，而不是 LLM 返回层。

### 可能原因

请重点排查：

1. 消息气泡 QLabel 设置了固定高度，导致长文本被裁剪。
2. 文本组件设置了 elide mode，自动把超长文本显示成 `...`。
3. HTML / Markdown 渲染时容器高度没有根据内容自适应。
4. 外层滚动区域没有重新计算布局高度。
5. 气泡最大高度被限制，超过部分不可见。
6. 使用了 `setFixedHeight` 或错误的 size policy。
7. 流式输出结束后没有触发 resize / relayout。
8. 消息保存的是完整内容，但 UI 加载历史时只取了 summary/snippet。

### 预期行为

聊天消息必须完整显示。

如果内容很长：

- 气泡高度应该自动增加。
- 聊天区域应该可以滚动查看完整内容。
- 不应该自动截断成省略号。
- 如果要做折叠功能，也必须有“展开全文”按钮，而不是静默截断。

### 修复要求

1. 检查消息气泡组件，移除不必要的固定高度。
2. 文本组件启用 word wrap。
3. 消息气泡高度根据内容自适应。
4. 外层 scroll area 在消息更新后正确滚动到底部。
5. 历史加载时必须加载完整 content，不要加载 summary。
6. 如果当前 UI 为了历史列表使用了 snippet，不要复用 snippet 作为聊天正文。
7. 流式输出时，每次追加 token 后要触发布局刷新；输出完成后再做一次最终 resize。

### 验收标准

- 长回答不会出现无意义省略号。
- 多行 Markdown / 列表内容完整显示。
- 历史会话恢复后，长消息仍完整显示。
- 聊天区域可以通过滚动查看完整内容。

## Bug 6：会话历史保存或读取逻辑错误，导致当前 session 的上下文和真实提问不匹配

### 现象描述

用户明确说明：当前 session 中只问了两个问题：

1. `你好，我是谁`
2. `你是谁`

但是当用户继续问：

```txt
我刚刚都问了什么问题
```

AI 的回答却提到了完全不相关的问题，例如：

```txt
看看桌面上简历文件夹里面都有哪些文件
你搞错了，里面不都是陈金保开头的吗
```

这说明当前会话上下文混入了其他 session 的历史，或者读取了错误的历史文件。

这是一个严重问题，因为它会让 AI 的短期记忆完全错乱。

### 复现步骤

1. 新建一个聊天。
2. 只发送：`你好，我是谁`。
3. 再发送：`你是谁`。
4. 再发送：`我刚刚都问了什么问题`。
5. 观察 AI 是否只回答这两个问题。
6. 当前实际结果：AI 回答了其他历史会话中的内容。

### 预期行为

当前 session 的上下文只能包含当前 session 的 messages。

除非明确设计了长期记忆系统，否则不能把其他历史会话直接混入当前对话上下文。

正确回答应该类似：

```txt
你刚刚问了两个问题：
1. 你好，我是谁
2. 你是谁
```

### 可能原因

请重点排查：

1. 发送给 LLM 的 messages 使用了全局历史，而不是当前 session messages。
2. 历史窗口点击某个 session 后，没有正确切换 current_session_id。
3. 新建聊天没有创建真正新的 session_id，仍然复用了旧 session。
4. 保存消息时写到了错误 session 文件。
5. 加载消息时按时间读取了最近全部 messages，而不是当前 session。
6. summary / memory 模块把其他会话内容注入到了当前 prompt。
7. 当前聊天标题更新了，但 session_id 没变。
8. UI 显示的是一个 session，后台 LLM 调用用的是另一个 session。

### 修复要求

1. 明确 `current_session_id` 的生命周期。
2. 新建聊天时必须生成新的 session_id。
3. 当前聊天窗口只显示当前 session 的 messages。
4. 发送给 LLM 的上下文只来自当前 session 的 messages。
5. 历史会话点击恢复时，必须同时更新：
   - current_session_id
   - UI messages
   - backend conversation state
6. 不允许默认把所有历史会话拼进当前 prompt。
7. 如果有长期记忆注入，需要单独标记为 memory，不得伪装成当前对话历史。
8. 添加 debug 日志：每次发请求前记录当前 session_id 和 messages 数量，但不要记录敏感内容。

### 建议调试日志

```txt
[chat] current_session_id=session_20260428_161000
[chat] request_messages_count=4
[chat] history_source=data/conversations/session_20260428_161000.json
```

禁止打印完整 API Key。

### 验收标准

- 新建聊天后，只问 A、B，再问“我刚刚问了什么”，只能回答 A、B。
- 切换历史会话后，聊天窗口显示的是被点击会话的内容。
- 切回新会话后，不会混入其他会话内容。
- LLM 请求日志中的 session_id 与 UI 当前标题对应。

## Bug 7：用户需要能自行调整聊天窗口大小

### 现象描述

当前聊天窗口大小看起来接近固定，用户无法自由调整窗口宽高，导致：

1. 长回答阅读困难。
2. 输入框空间不够。
3. 任务窗口或设置窗口容易遮挡主窗口。
4. 不同屏幕分辨率下体验不好。

### 预期行为

用户应该可以拖动窗口边缘调整聊天窗口大小。

调整后：

- 标题栏保持正常。
- 消息区域自动拉伸。
- 输入框区域固定在底部。
- Send 按钮仍在右下角。
- 滚动条正常工作。
