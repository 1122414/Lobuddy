# Lobuddy 陪伴体验增强阶段开发计划

> 交付目标：在不重构现有架构、不破坏已有聊天 / 历史 / 设置 / 自定义宠物功能的前提下，把 Lobuddy 从“聊天窗口 + 宠物图标”升级为“有反馈、有时间感、有陪伴感、可长期扩展的桌宠助手”。

---

## 0. 当前阶段定位

### 0.1 本阶段要解决什么

当前 Lobuddy 已经具备：

- 桌宠主体 UI
- 聊天窗口
- 历史会话
- 设置窗口
- 自定义宠物图片 / GIF
- 基础 Agent 对话能力

本阶段重点不是继续堆 Agent 工具能力，而是提升桌宠体验：

1. 用户点击宠物时，宠物应该有反馈。
2. 宠物卡片应该有轻量时间感，例如当前日期时间。
3. 用户对话应该有时间线，而不是普通聊天框。
4. 右侧应该有可跳转的对话节点，方便用户回看。
5. 宠物应该具备简单状态，例如陪你中、思考中、待机中。
6. 所有陪伴功能都必须可配置、可关闭、可扩展。

### 0.2 本阶段不解决什么

本阶段严禁做以下事情：

- 不重写整个 UI。
- 不替换 PySide6 技术栈。
- 不引入大型动画库、前端框架或复杂依赖。
- 不重构 Agent 核心执行链路。
- 不修改 LLM 调用协议，除非当前代码必须读取新增配置。
- 不新增复杂多 Agent 能力。
- 不做联网天气、日历、任务管理等大功能。
- 不做复杂宠物养成系统，例如背包、商店、战斗、皮肤商城。
- 不做声音系统，只预留接口。
- 不做自动长期记忆写入，只预留记忆卡片入口。
- 不做数据库迁移，继续使用当前历史存储方式，必要时只做兼容字段扩展。

---

## 1. 全局开发原则

### 1.1 代码原则

1. 小步修改，避免一次性大改。
2. UI 组件独立封装，不要继续把全部逻辑堆进主窗口。
3. 所有新增功能必须从配置读取开关。
4. 所有新增文案、默认值、时间格式、动画参数必须可配置。
5. 当前阶段的“常量配置”统一放入 `.env` 或现有 env 配置体系中，不允许散落硬编码。
6. UI 组件只负责展示，不直接负责业务状态判断。
7. 状态判断交给 manager / controller 层。
8. 新增功能必须不影响已有历史、设置、自定义图片、聊天输入、滚动条。

### 1.2 UI 原则

保持当前 Lobuddy 的视觉语言：

- 橙色主色
- 奶油色背景
- 圆角卡片
- 轻量阴影
- 低打扰陪伴感
- 不要做赛博朋克、纯黑、强科技感风格

### 1.3 用户体验原则

1. 陪伴功能要温和，不要频繁弹窗。
2. 不要阻塞用户操作。
3. 动画不能影响聊天输入。
4. 时间线和提示气泡不能遮挡主要内容。
5. 所有主动提醒必须可以关闭。

## 2. 配置与 env 规范

### 2.1 必须先完成的基础任务

在实现功能前，先整理新增配置项。所有新增常量必须进入 `.env` 或当前项目已有 env 配置系统。

如果项目已经有 `ConfigManager` / `SettingsManager`，请复用它；如果没有，请新增一个轻量配置入口，但不要大改架构。

### 2.2 新增 env 配置项

请在 `.env.example`、配置读取代码、设置页同步逻辑中补充以下字段。

LOBUDDY_PET_CLOCK_HOVER_FULL_FORMAT=true

# 聊天消息时间

LOBUDDY_CHAT_TIME_DIVIDER_GAP_MINUTES=5

### 2.3 配置读取要求

必须支持：

1. 程序启动时从 `.env` 读取默认值。
2. 设置页打开时从当前配置对象读取值。
3. 设置页保存时写回 `.env` 或当前持久化配置文件。
4. 保存后当前运行时立即生效，不要求重启。
5. 不允许 UI 中写死默认值。
6. 不允许不同模块各自读取不同配置源。

### 2.4 设置页要求

当前设置页已有多个 tab，请在合适位置补充：

- 陪伴设置
- 聊天设置
- 高级设置

如果当前 UI 不方便新增 tab，可以先放入现有“陪伴设置”和“高级设置”中，但代码上要预留分组。

# 5. P0 任务 1：统一新增配置接入

## 5.1 目标

新增功能全部依赖 env 配置，避免后续功能开关散落在代码中。

## 5.2 需要做什么

1. 增加陪伴功能配置字段。
2. 更新 `.env.example`。
3. 更新设置读取逻辑。
4. 更新设置保存逻辑。
5. 设置页增加对应开关。
6. 保存后能立即影响当前 UI。

## 5.3 严禁做什么

- 不允许每个 widget 自己直接 `os.getenv()`。
- 不允许把默认值写死在 UI 文件里。
- 不允许保存设置后必须重启才能生效。
- 不允许把新增配置和 LLM API Key 逻辑混在一起。

## 5.4 建议实现

新增配置对象，例如：

```python
@dataclass
class CompanionSettings:
    pet_click_feedback_enabled: bool
    pet_click_cooldown_ms: int
    pet_bubble_duration_ms: int
    pet_clock_enabled: bool
    pet_clock_show_seconds: bool
    chat_message_time_enabled: bool
    chat_time_divider_enabled: bool
    chat_time_divider_gap_minutes: int
    conversation_timeline_enabled: bool
    pet_state_enabled: bool
```

配置读取由现有 `ConfigManager` 统一完成。

## 5.5 验收标准

- 修改 `.env` 后，启动程序能读取新值。
- 设置页保存后，`.env` 或当前配置文件能更新。
- 设置页重新打开后显示保存后的值。
- 关闭某个开关后，对应功能立即隐藏或停止。

# 6. P0 任务 2：宠物点击反馈

## 6.1 目标

用户点击宠物主体时，随机触发轻量反馈，让宠物像“有反应”。

## 6.2 需要做什么

点击宠物图片区域时，随机触发以下反馈之一：

1. 轻微跳动。
2. 轻微缩放。
3. 宠物旁边弹出短句气泡。
4. 宠物状态文字短暂变化。
5. 连续点击达到阈值后触发小彩蛋。

## 6.3 建议反馈文案

文案也要放入可配置常量中，至少集中在一个文件，不要散落。

```txt
我在呢～
今天也辛苦啦！
摸摸头成功！
要不要休息一下？
我会陪着你的～
别戳啦，会害羞的！
```

如果必须严格遵守“常量放 env”，可将文案用分隔符放入 env：

```env
LOBUDDY_PET_CLICK_MESSAGES=我在呢～|今天也辛苦啦！|摸摸头成功！|要不要休息一下？|我会陪着你的～
LOBUDDY_PET_CLICK_EASTER_EGG_MESSAGE=别戳啦，会害羞的！
```

## 6.4 技术要求

1. 使用 PySide6 自带 `QPropertyAnimation`。
2. 不引入大型动画库。
3. 加点击冷却，避免动画堆积。
4. 不能影响拖动宠物窗口。
5. 如果当前点击事件同时承担拖动逻辑，需要区分 click 和 drag：
   - 鼠标按下记录位置。
   - 鼠标移动超过阈值视为拖动。
   - 鼠标释放且移动距离小于阈值才视为点击。

## 6.5 预留接口

后续可以接入：

- 不同宠物情绪包
- 点击音效
- 表情切换
- 触摸区域，例如头部、身体、按钮不同反馈

当前只预留字段，不实现音效。

## 6.6 验收标准

- 点击宠物有随机反馈。
- 连续点击不会导致卡顿。
- 拖动宠物不会误触发大量点击反馈。
- 设置中关闭后，点击不再触发反馈。

# 7. P0 任务 3：宠物右下角时钟

## 7.1 目标

在宠物卡片右下角显示当前日期和时间，增强桌面陪伴感。

## 7.2 显示规则

默认显示：

```txt
04/28 20:35
```

如果开启秒数：

```txt
04/28 20:35:21
```

鼠标悬停 tooltip：

```txt
2026年4月28日 20:35:21
```

## 7.3 UI 要求

1. 放在宠物卡片右下角。
2. 不遮挡宠物主体。
3. 不遮挡等级和经验条。
4. 使用小圆角胶囊样式。
5. 颜色沿用橙色 / 奶油白。
6. 字体小一点，不抢主视觉。

## 7.4 技术要求

1. 新增独立 `PetClockWidget`。
2. 使用 `QTimer` 刷新。
3. 默认每 30 秒刷新。
4. 开启秒数时每 1 秒刷新。
5. 时间格式从配置读取。

## 7.5 严禁做什么

- 不要在主窗口里直接写死 timer。
- 不要因为时钟导致整个宠物 widget 高频 repaint。
- 不要显示过大时间，避免变成时钟软件。

## 7.6 验收标准

- 宠物右下角显示时间。
- 开关关闭后隐藏。
- 开启秒数后秒数正常刷新。
- 切换宠物图片 / GIF 后，时钟仍然位置正常。

# 8. P0 任务 4：聊天消息时间显示

## 8.1 目标

每条消息要具备真实时间戳，UI 中可以展示发送时间，历史恢复后时间也正确。

## 8.2 数据结构要求

当前消息结构需要包含或兼容：

```json
{
  "id": "message_xxx",
  "role": "user | assistant | system",
  "content": "...",
  "timestamp": "2026-04-28T20:35:21+08:00",
  "metadata": {}
}
```

如果历史旧数据没有 `timestamp`，加载时需要兼容：

- 可以使用会话更新时间。
- 可以标记为未知时间。
- 不允许因为缺字段导致历史加载失败。

## 8.3 UI 显示规则

用户消息和助手消息都显示小时间：

```txt
20:35
```

建议放在气泡右下角或气泡下方的小字区域。

## 8.4 技术要求

1. 发送用户消息时立即生成 timestamp。
2. 接收助手回复时生成 timestamp。
3. 保存历史时写入 timestamp。
4. 加载历史时读取 timestamp。
5. 时间格式使用 `time_format.py` 统一处理。
6. 开关关闭时隐藏小时间，但数据仍然保存。

## 8.5 严禁做什么

- 不要只在 UI 临时生成当前时间。
- 不要历史加载时全部显示为当前时间。
- 不要因为旧历史缺 timestamp 就崩溃。

## 8.6 验收标准

- 新发消息显示时间。
- 助手回复显示时间。
- 重启后历史消息时间仍然正确。
- 关闭显示时间后，UI 不显示但数据仍保存。

# 9. P0 任务 5：对话时间分隔条

## 9.1 目标

在聊天流中插入时间分隔条，让会话有明确时间段。

## 9.2 显示规则

当相邻消息间隔超过配置值，默认 5 分钟，插入分隔条：

```txt
今天 20:35
```

跨天时显示：

```txt
2026年4月28日 周二
```

## 9.3 技术要求

1. 新增 `MessageTimeDivider` widget。
2. 渲染聊天消息时判断是否需要插入分隔条。
3. 历史加载时也执行同样逻辑。
4. gap 分钟数从 env 读取。
5. 用户关闭后不显示分隔条。

## 9.4 边界处理

1. 第一条消息可以显示一个时间分隔条。
2. 缺失 timestamp 的旧消息，不插入或显示“较早”。
3. 新建聊天清空分隔条。

## 9.5 验收标准

- 间隔超过 5 分钟的消息中间出现时间条。
- 同一段连续对话不会被过多时间条打断。
- 历史恢复后时间条正确。

# 10. P1 任务 6：右侧对话时间线

## 10.1 目标

在聊天窗口右侧增加细竖向时间线。每次用户发送消息时，在时间线上新增一个点。用户 hover 可以看到消息缩略，点击可以跳转到对应消息。

## 10.2 基础行为

1. 用户发送消息时新增一个 timeline dot。
2. 每个 dot 绑定对应 user message 的 `message_id`。
3. hover dot 显示：

```txt
20:35 告诉我今天广州的天气...
```

4. 点击 dot 后，聊天滚动区域跳转到对应消息气泡。
5. 当前可视区域附近的 dot 可以高亮。
6. 加载历史会话后，时间线根据用户消息恢复。

## 10.3 UI 要求

1. 时间线在聊天窗口右侧。
2. 不能覆盖系统滚动条。
3. 可以放在滚动区域内侧右边，也可以放在滚动区域外侧独立窄栏。
4. 宽度控制在 12px 到 24px。
5. 点不要过大。
6. hover tooltip 使用圆角卡片风格。

## 10.4 技术要求

1. 新增 `ConversationTimelineWidget`。
2. 时间线数据来自当前 session message list，不要从 UI 文本反向解析。
3. 每个 user message 必须有稳定 `message_id`。
4. 聊天 bubble widget 需要能按 message_id 定位。
5. 需要维护 `message_id -> bubble_widget` 映射。
6. 点击 dot 使用滚动区域的 `ensureWidgetVisible()` 或等效方法跳转。
7. 如果消息很多，需要做点位压缩。

## 10.5 点位压缩规则

当用户消息过多时，不要所有点都挤在一起。

建议 MVP：

1. 如果两个点之间小于 `LOBUDDY_CONVERSATION_TIMELINE_MIN_DOT_GAP_PX`，则合并视觉点。
2. 合并点 hover 时显示最近一条用户消息。
3. 后续再扩展为显示多条列表。

## 10.6 严禁做什么

- 不要替换当前聊天滚动组件。
- 不要破坏原有滚动条。
- 不要用 OCR / 截图方式定位消息。
- 不要从 HTML / QLabel 文本里反向解析消息。

## 10.7 预留接口

后续可以支持：

- 按时间段折叠
- 搜索结果标记
- 收藏消息标记
- Agent 工具调用节点标记
- 错误 / 成功任务节点标记

当前只做用户消息点。

## 10.8 验收标准

- 发送用户消息后右侧出现点。
- hover 点显示用户消息摘要。
- 点击点能跳转。
- 历史会话加载后时间线恢复。
- 关闭开关后时间线隐藏。

# 11. P1 任务 7：宠物状态系统

## 11.1 目标

让宠物根据当前行为显示不同状态，而不是一直是 `OK`。

## 11.2 状态列表

建议最小状态：

```txt
Idle        待机中
Listening   倾听中
Thinking    思考中
Working     工作中
Happy       开心
Sleepy      困困
Error       需要看看
```

## 11.3 触发规则

| 事件 | 状态 |
|---|---|
| 用户正在输入 | Listening |
| 用户发送消息后等待回复 | Thinking |
| Agent 工具执行中 | Working |
| 点击宠物反馈 | Happy |
| 长时间无交互 | Idle |
| 夜间时间段 | Sleepy |
| 请求失败 / API key 错误 | Error |

## 11.4 技术要求

1. 新增 `PetStateManager`。
2. UI 只订阅状态变化，不直接判断复杂规则。
3. 状态变更需要有优先级，例如 Error > Working > Thinking > Happy > Sleepy > Idle。
4. 临时状态需要自动恢复，例如 Happy 2 秒后回到之前状态。
5. 状态文案和阈值来自 env。

## 11.5 env 扩展

```env
LOBUDDY_PET_STATE_TEXT_IDLE=待机中
LOBUDDY_PET_STATE_TEXT_LISTENING=倾听中
LOBUDDY_PET_STATE_TEXT_THINKING=思考中
LOBUDDY_PET_STATE_TEXT_WORKING=工作中
LOBUDDY_PET_STATE_TEXT_HAPPY=开心
LOBUDDY_PET_STATE_TEXT_SLEEPY=困困
LOBUDDY_PET_STATE_TEXT_ERROR=需要看看
```

## 11.6 严禁做什么

- 不要让状态系统影响真实任务执行。
- 不要为了状态系统改 Agent 运行流程。
- 不要把状态判断散落在多个 UI 事件里。

## 11.7 验收标准

- 发送消息后状态变为思考中。
- 回复完成后恢复正常状态。
- 点击宠物时短暂变开心。
- 晚上时间可显示困困。
- 关闭开关后状态固定或隐藏。

# 17. 手动测试清单

## 17.1 配置测试

1. 修改 `.env` 中 `LOBUDDY_PET_CLOCK_ENABLED=false`。
2. 启动程序。
3. 确认宠物时钟不显示。
4. 在设置页打开该开关并保存。
5. 确认时钟立即显示。
6. 重启后仍然显示。

## 17.2 宠物点击反馈测试

1. 点击宠物图片。
2. 确认出现动画或气泡。
3. 快速连续点击 10 次。
4. 确认不卡顿、不报错。
5. 拖动宠物。
6. 确认拖动不会疯狂触发点击反馈。

## 17.3 时钟测试

1. 查看宠物右下角是否显示月、日、时、分。
2. 打开秒数显示。
3. 确认秒数刷新。
4. hover 时钟查看完整时间。
5. 切换宠物 GIF 后确认不遮挡。

## 17.4 消息时间测试

1. 发送用户消息。
2. 等待助手回复。
3. 确认两条消息都有时间。
4. 重启程序加载历史。
5. 确认时间仍然是原时间，而不是当前时间。

## 17.5 时间分隔条测试

1. 构造两条间隔超过 5 分钟的消息。
2. 加载历史。
3. 确认中间出现时间分隔条。
4. 连续短时间消息不应每条都插入分隔条。

## 17.6 右侧时间线测试

1. 连续发送 5 条用户消息。
2. 确认右侧出现 5 个点或合理压缩点。
3. hover 每个点能看到消息缩略。
4. 点击点能跳转到对应用户消息。
5. 切换历史会话后，时间线更新。
6. 新建聊天后，时间线清空。

## 17.7 宠物状态测试

1. 未操作一段时间，显示待机。
2. 输入文字时显示倾听中。
3. 发送消息等待回复时显示思考中。
4. 任务执行时显示工作中。
5. 点击宠物短暂显示开心。
6. API 错误时显示需要看看。
