# 4.30 主题升级方案：宠物 UI 自动生成主题 + 用户可编辑主题系统

## 1. 背景与目标

当前需求是：用户上传宠物 UI 后，系统能够自动分析该 UI 的颜色配置，并生成一套适配 Lobuddy 产品的主题颜色。同时，这套主题需要永久保存在用户的主题配色中，用户可以像使用预制主题一样启用它，也可以删除、编辑、复制自己的主题。

本次升级的核心目标不是简单地“从图片中提取几个颜色”，而是建立一套完整的用户主题机制：

```text
用户上传宠物 UI
↓
系统分析颜色
↓
生成完整主题草稿
↓
用户编辑和预览
↓
保存到“我的主题”
↓
和预制主题一样全局应用
```

核心原则：

> 自动生成负责给用户一个好起点，主题编辑器负责让用户拥有最终控制权。

---

## 2. 总体方案

建议将该能力设计成“从宠物 UI 自动生成主题”的主题系统扩展，而不是单独给某几个页面做颜色适配。

完整流程如下：

```text
用户上传宠物 UI
↓
前端或后端提取图片主色
↓
生成主题候选色板
↓
做可读性、对比度、浅色/深色适配
↓
映射成完整 Design Token
↓
展示主题预览和编辑页面
↓
用户确认或修改颜色
↓
保存为用户自定义主题
↓
写入数据库
↓
通过全局 Theme Provider 应用
```

关键点：

1. 自动生成的主题不能直接作为最终结果强制保存。
2. 自动生成结果应该先作为“主题草稿”。
3. 用户可以修改自动生成的颜色。
4. 保存后的用户主题和预制主题必须使用同一套 Theme Token。
5. 所有组件都应该通过统一的语义 Token 使用颜色，而不是直接写死颜色值。

---

## 3. 主题系统设计原则

### 3.1 预制主题和用户主题统一管理

不要把预制主题和用户生成主题做成两套逻辑。

推荐统一成：

```text
主题列表 = 预制主题 + 用户自定义主题
```

主题来源可以区分为：

```text
preset：系统预制主题
pet-ui-generated：由宠物 UI 自动生成
manual：用户手动创建或编辑
```

但是在应用层面，它们应该完全一致：

```text
选择主题
↓
读取 Theme Tokens
↓
写入全局 CSS Variables
↓
所有页面和组件自动生效
```

这样可以避免后续出现：

```text
预制主题一套逻辑
用户主题一套逻辑
宠物生成主题又一套逻辑
```

否则主题系统后期会越来越难维护。

---

### 3.2 所有主题都转成统一 Theme Tokens

不管主题来自哪里，都必须转成统一结构。

建议至少包括这些颜色 Token：

```text
primary：主色
primaryHover：主色悬浮态
primaryActive：主色按下态
secondary：辅助色
accent：强调色
background：页面背景色
surface：容器背景色
card：卡片背景色
textPrimary：一级文字
textSecondary：二级文字
textMuted：弱化文字
border：边框色
divider：分割线色
success：成功色
warning：警告色
danger：错误色
info：信息色
onPrimary：主色上的文字色
onAccent：强调色上的文字色
```

用户上传图片得到的不是一个颜色，而是一整套可应用到产品所有地方的主题 Token。

---

## 4. 宠物 UI 颜色分析思路

上传宠物 UI 后，可以分两层分析。

### 4.1 第一层：提取图片主色

可以从图片中提取：

```text
主色 primary
辅助色 secondary
点缀色 accent
背景候选 background
中性色 neutral
高亮色 highlight
```

常见方法包括：

```text
K-Means 聚类
Median Cut
Octree Quantization
Vibrant.js
ColorThief
```

如果希望快速落地，前端可以使用 Vibrant.js 或 ColorThief。

如果希望结果更稳定，建议在后端或 AI 服务中处理，方便加入更多校验和优化逻辑。

---

### 4.2 第二层：颜色语义映射

不要直接把图片中面积最大的颜色当主题主色。

更合理的做法是根据颜色特征进行语义映射：

```text
面积较大且明度较高的浅色 → background / surface
饱和度较高但不刺眼的颜色 → primary / accent
较深颜色 → textPrimary
中等灰度颜色 → border / textSecondary
柔和辅助色 → secondary
```

这样可以避免出现这些问题：

```text
背景色过深，文字看不清
主色太亮，按钮刺眼
图片颜色很好看，但放到 UI 上不可用
颜色之间缺少层级
```

---

## 5. 主题生成策略

### 5.1 不只保存单个颜色，而是生成色阶

系统从宠物 UI 中识别出主色后，不应该只保存一个主色值，而是生成完整色阶。

例如：

```text
primary-50
primary-100
primary-200
primary-300
primary-400
primary-500
primary-600
primary-700
primary-800
primary-900
```

这样后续可以用于：

```text
按钮默认态
按钮 hover 态
按钮 pressed 态
浅色背景
边框
标签背景
选中态
高亮提示
```

---

### 5.2 使用语义 Token，而不是直接使用色值

组件里不要直接使用图片提取出的颜色。

错误思路：

```text
按钮直接使用 #FF9F7A
卡片直接使用 #FFF3EE
```

正确思路：

```text
按钮使用 primary
卡片使用 card
页面使用 background
文字使用 textPrimary
边框使用 border
```

这样无论用户启用预制主题还是宠物生成主题，组件都不需要关心具体颜色值。

---

### 5.3 自动补全关联颜色

用户修改一个颜色时，系统应该自动补全相关颜色。

例如用户修改“主色”后，系统可以自动更新：

```text
主色
主色 Hover
主色 Pressed
主色浅色背景
主色边框
主色上的文字色
```

不建议要求用户手动修改所有状态色，否则编辑成本过高。

可以提供两种模式：

```text
自动联动：修改主色后，系统自动生成关联色
解除关联：高级用户可以单独编辑每个颜色
```

---

## 6. 用户可编辑主题能力

### 6.1 自动生成只是草稿

用户上传宠物 UI 后，系统生成的是主题草稿，而不是直接保存的最终主题。

推荐流程：

```text
上传宠物 UI
↓
系统分析颜色
↓
生成主题草稿
↓
进入主题编辑器
↓
用户修改颜色
↓
实时预览
↓
保存到“我的主题”
```

这样用户可以对自动生成结果进行控制，而不是被动接受系统生成的配色。

---

### 6.2 主题编辑器分为简单模式和高级模式

不建议直接展示几十个颜色字段，会让普通用户觉得复杂。

建议分成两层。

#### 简单模式

适合普通用户，只展示最容易理解的颜色项：

```text
主色
背景色
卡片色
文字色
按钮色
强调色
边框色
错误色
成功色
```

#### 高级模式

适合希望精细调整的用户：

```text
Primary 色阶
Secondary 色阶
Accent 色阶
Background / Surface / Card
Text Primary / Text Secondary / Text Muted
Border / Divider
Success / Warning / Danger / Info
```

默认展示简单模式，高级模式可以折叠。

---

### 6.3 编辑时提供实时预览

主题编辑页面建议包含三个区域：

```text
左侧：颜色配置
中间：真实组件预览
右侧：主题信息和质量提示
```

预览区域应该覆盖真实产品中常见组件：

```text
首页背景
宠物卡片
按钮
输入框
Tab
标签
聊天气泡
弹窗
表单错误提示
底部导航
```

用户修改一个颜色后，预览区立即变化。

建议编辑时只影响预览区，不直接影响全局主题。用户点击“保存并应用”后，再全局生效。

这样可以避免用户在编辑过程中把整个产品颜色改乱。

---

### 6.4 显示颜色影响范围

用户修改某个颜色时，应该告诉他这个颜色会影响哪里。

例如修改“主色”时：

```text
主色会影响：
按钮背景、选中状态、链接、重点标签、开关开启状态
```

修改“背景色”时：

```text
背景色会影响：
页面底色、空状态背景、部分容器外层区域
```

这样用户知道自己正在改什么，不会盲目调整。

---

### 6.5 支持撤销、恢复和复制

用户编辑主题时，建议支持：

```text
撤销
重做
恢复为自动生成版本
恢复为上次保存版本
复制当前主题
```

尤其是“恢复为自动生成版本”很重要。

因为用户可能越调越乱，需要一个安全回退点。

---

## 7. 可读性和对比度校验

自动生成主题和用户编辑主题都必须做可读性校验。

至少需要检查：

```text
textPrimary / background
textSecondary / background
onPrimary / primary
onAccent / accent
button text / button background
input text / input background
link / background
error / background
```

建议分级提示：

```text
绿色：可读性良好
黄色：建议优化
红色：可能难以阅读
```

不要一开始就强行阻止用户保存，除非严重不可读。

更好的方式是提供修复建议：

```text
当前按钮文字与按钮背景对比度过低
建议将按钮文字改为白色或深色
```

并提供操作：

```text
一键提高可读性
使用系统推荐文字色
```

这样既保证可用性，又不会让用户觉得被系统限制。

---

## 8. 主题状态设计

建议区分三个状态：

```text
草稿主题
已保存主题
当前启用主题
```

### 草稿主题

用户正在编辑，还没有保存。

### 已保存主题

已经存在于“我的主题”中，可以再次编辑、复制、删除。

### 当前启用主题

正在全局生效的主题。

推荐交互：

```text
编辑时只影响预览
保存后进入“我的主题”
点击“保存并应用”后才影响全局
```

---

## 9. 数据持久化设计思路

用户生成的主题需要绑定账号保存，而不是只存在本地。

建议保存这些信息：

```text
主题 ID
用户 ID
主题名称
主题来源
源图片地址
完整主题 JSON
是否当前启用
是否删除
是否被用户编辑过
创建时间
更新时间
```

用户删除主题时，建议使用软删除，而不是物理删除。

原因：

```text
可以支持恢复
可以避免历史引用失效
方便做数据审计
```

用户编辑后的主题应该保存完整主题 Token，而不是只保存用户修改过的几个颜色。

可以理解为：

```text
自动生成主题草稿
+
用户修改覆盖
+
系统补全和校验
=
最终完整主题
```

保存后的主题可以标记为：

```text
来源：宠物 UI 自动生成
状态：已被用户手动调整
```

这样后续可以展示：

```text
由宠物 UI 生成，已手动调整
```

---

## 10. 用户主题管理能力

在“设置 → 外观”中建议分为：

```text
预制主题
我的主题
从宠物 UI 生成主题
```

用户生成的主题支持：

```text
启用
重命名
编辑颜色
复制
删除
恢复自动生成配色
```

预制主题建议支持：

```text
启用
复制为我的主题后编辑
```

预制主题不建议直接编辑或删除。

这样可以保证预制主题稳定，同时给用户足够的自定义空间。

## 12. MVP 建议

第一版不要做太大，建议先完成核心闭环。

MVP 范围：

```text
1. 用户上传宠物 UI
2. 系统提取 5 个主要颜色
3. 自动生成 1 套浅色主题草稿
4. 用户可以修改主色、背景色、文字色、按钮色、卡片色、边框色
5. 修改后预览区实时变化
6. 系统做基础对比度校验
7. 用户点击保存
8. 保存到“我的主题”
9. 用户可以启用、重命名、编辑、复制、删除
10. 预制主题和用户主题统一通过 Theme Tokens 全局应用
```

第一版暂时不建议做：

```text
复杂 AI 命名
主题分享
多套风格同时生成
深色模式自动生成
主题评分系统
局部组件风格生成
```

这些可以放到后续版本。

---

## 13. 后续可升级方向

### 13.1 一次生成多个风格

用户上传同一张宠物 UI 后，可以生成多个版本：

```text
柔和版：适合长时间使用
鲜艳版：更活泼
高级版：降低饱和度，更克制
```

这样用户选择空间更大。

---

### 13.2 支持浅色和深色双模式

未来可以让一次生成同时得到：

```text
light theme
 dark theme
```

暗色模式不要简单反转颜色，而应该重新生成：

```text
深色背景
低亮度 surface
高对比文字
保留宠物 UI 的主色作为 accent
```

---

### 13.3 根据宠物类型生成主题名称

可以结合图像识别结果生成更有情绪价值的名称：

```text
金毛暖阳
布偶奶蓝
柴犬焦糖
黑猫月影
鹦鹉森绿
```

这会让功能更有记忆点。

---

### 13.4 主题质量评分

可以给主题增加内部评分或用户可见提示：

```text
可读性评分
视觉协调度
品牌一致性
长时间使用舒适度
```

用户侧不一定要展示复杂分数，可以用简单文案：

```text
这个主题适合长时间使用
这个主题按钮文字略浅，建议优化
```

---

### 13.5 色彩锁定

用户可以锁定某些颜色不被系统自动修改。

例如：

```text
锁定主色
锁定背景色
锁定文字色
```

当用户点击“一键优化”时，系统只调整未锁定的颜色。

---

### 13.6 主题分享和导入

后续可以支持：

```text
生成分享链接
导出主题码
导入他人主题
复制主题 JSON
```

这可以让用户之间分享宠物主题，提高传播性。

---

### 13.7 从当前主题继续调整

用户可以基于已有主题继续生成：

```text
用当前主题作为基础
重新从宠物 UI 提取颜色
只替换主色
只替换背景色
只替换强调色
```

这比每次从零开始更灵活。

---

### 13.8 局部风格 Token

颜色稳定后，可以继续扩展到：

```text
圆角大小
阴影柔和度
按钮风格
卡片密度
插画氛围
动效节奏
```

但建议第二阶段再做，不要放进 MVP。

---

## 14. 风险点与注意事项

### 14.1 图片颜色不等于可用 UI 颜色

宠物 UI 图片里的颜色可能很好看，但直接放到产品界面中不一定可用。

必须经过：

```text
语义映射
色阶生成
对比度校验
可读性修正
```

---

### 14.2 不要让用户编辑项过多

如果一开始暴露几十个颜色字段，普通用户会觉得复杂。

建议：

```text
默认简单模式
高级模式折叠
提供系统推荐
提供一键修正
```

---

### 14.3 不要让编辑过程直接影响全局

用户编辑颜色时，建议只影响预览区域。

用户点击“保存并应用”后，再影响全局。

这样更安全。

---

### 14.4 主题必须全局统一生效

按钮、卡片、输入框、弹窗、导航、聊天气泡等都要使用同一套 Theme Token。

如果部分页面仍然写死颜色，会导致用户主题体验不完整。

---

## 15. 最终推荐方案

本次 4.30 主题升级建议采用以下方案：

```text
1. 建立统一 Theme Token 体系
2. 预制主题和用户主题统一走同一套应用逻辑
3. 用户上传宠物 UI 后生成主题草稿
4. 用户进入主题编辑器调整颜色
5. 编辑器提供实时组件预览
6. 系统自动做对比度和可读性校验
7. 用户保存后进入“我的主题”
8. 用户主题支持启用、编辑、复制、重命名、删除
9. 预制主题不可直接编辑，但可以复制后编辑
10. 后续再扩展深色模式、多风格生成、主题分享和主题评分
```

最重要的产品原则是：

> 所有主题，不管是预制的、用户手动创建的，还是宠物 UI 自动生成的，都必须转换成同一套 Theme Tokens。

只要 Theme Token 统一，后续的保存、删除、编辑、预览、启用、全局应用、暗色模式扩展都会非常顺。
