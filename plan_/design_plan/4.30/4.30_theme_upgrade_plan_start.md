# 4.30 主题升级方案 - 可行性分析与实施计划

## 1. 现状分析

### 1.1 当前主题系统架构

| 组件 | 位置 | 说明 |
|------|------|------|
| ThemeColors | `ui/theme.py` | 40+ 设计 token 的 dataclass |
| ThemeManager | `ui/theme.py` | 单例，`theme_changed` 信号传播 |
| 预制主题 | `ui/theme.py` | 4 套：COZY_ORANGE, SAKURA_PINK, MINT_GREEN, NIGHT_COMPANION |
| 主题持久化 | `core/storage/settings_repo.py` | SQLite `app_settings` 表 key-value 存储 |
| 外观配置 | `core/models/appearance.py` | JSON 文件 `data/pet_appearance.json` |
| 资源管理 | `core/services/pet_asset_service.py` | 上传验证、复制、删除 |

### 1.3 关键差距

| 方案需求 | 当前状态 | 差距 |
|----------|----------|------|
| 统一 Theme Token | ✅ 已有基础 | 缺少 ~10 个 Token |
| 用户主题存储 | ❌ 无 | 需新建表或扩展现有存储 |
| 图片颜色提取 | ❌ 无 | 需引入库或实现算法 |
| 主题编辑器 UI | ❌ 无 | 需新建对话框 |
| 对比度校验 | ❌ 无 | 需实现 WCAG 算法 |
| 实时预览 | ⚠️ 部分 | ThemeManager 信号已支持，但编辑器需新建 |

## 2. 可行性评估

### 2.1 ✅ 高度可行

**统一 Theme Token 体系**
- `ThemeColors` 已有 40+ Token，结构清晰
- 只需补充 ~10 个缺失 Token（primary_hover, secondary, accent, card, divider, info, on_primary, on_accent）
- 所有组件通过 `ThemeManager.current` 获取颜色，改一处全局生效

**预制主题与用户主题统一管理**
- `ThemeManager` 已支持 `set_custom(overrides)` 模式
- 只需扩展存储：将用户主题完整 Token 存入 DB
- 切换逻辑：读取 Token → 调用 `apply_theme()` → 信号传播

**主题持久化**
- `SettingsRepository` 已有 `set_json_setting()` 方法
- 可直接用于存储用户主题 JSON
- 或新建 `user_themes` 表支持多主题管理

### 2.2 ⚠️ 中等可行（需评估）

**图片颜色提取**
- 方案提到 Vibrant.js（前端）或后端算法
- 项目已有 Pillow 依赖（用于图片分析），可用 Python 实现
- 推荐方案：使用 `colorthief-py` 库或自实现 K-Means
- 风险：提取质量依赖图片内容，需做语义映射和可读性修正

**主题编辑器 UI**
- 需新建 `ThemeEditorDialog`（类似 `SettingsWindow`）
- 工作量较大：颜色选择器、实时预览、对比度显示
- 可分阶段：先做简单模式（6-8 个核心颜色），后做高级模式

**对比度校验**
- WCAG 2.0 对比度算法公开可用
- Python 实现简单：`contrast_ratio(lum1, lum2)`
- 可在编辑时实时显示校验结果

### 2.3 ⚠️ 需谨慎评估

**从宠物 UI 自动生成主题**
- 图片颜色 ≠ 可用 UI 需色，需做语义映射
- 需处理：浅色/深色背景适配、文字可读性、交互态生成
- 建议：MVP 先做"提取 5 色 + 简单映射"，不做 AI 命名/多风格生成

**自动补全关联颜色**
- 修改主色后自动更新 hover/pressed/soft 等
- 需实现颜色变换函数（lighten/darken/saturate）
- 可选：先手动编辑，后加自动联动

## 3. 推荐实施路径

### Phase 1: Token 体系补全（1-2 天）

**目标**：补齐缺失 Token，确保所有组件使用统一 Token

**任务**：
1. 在 `ThemeColors` 中添加缺失 Token：
   ```python
   primary_hover: str      # 主色悬浮态
   primary_active: str     # 主色按下态
   secondary: str          # 辅助色
   accent: str             # 强调色
   card: str               # 卡片背景
   divider: str            # 分割线
   info: str               # 信息色
   on_primary: str         # 主色上的文字
   on_accent: str          # 强调色上的文字
   ```

2. 为 4 套预制主题补充新 Token 值

3. 更新 QSS 生成函数使用新 Token

4. 扫描所有 UI 文件，替换硬编码颜色为 Token 引用

**验证**：`ruff check .` + 手动测试 4 套主题切换

---

### Phase 2: 用户主题存储（1 天）

**目标**：支持保存/读取/删除用户自定义主题

**方案 A：扩展现有 `app_settings` 表**
```python
# 存储结构
key = "user_themes"
value = json.dumps({
    "theme_id_1": {
        "name": "我的主题 1",
        "source": "pet-ui-generated",
        "colors": { ... },  # 完整 ThemeColors dict
        "created_at": "...",
        "updated_at": "...",
        "is_active": False
    },
    ...
})
```

**方案 B：新建 `user_themes` 表（推荐）**
```sql
CREATE TABLE user_themes (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    source TEXT NOT NULL DEFAULT 'manual',  -- manual / pet-ui-generated
    colors_json TEXT NOT NULL,              -- 完整 ThemeColors JSON
    source_image_path TEXT,                 -- 来源图片路径（可选）
    is_active INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

**任务**：
1. 在 `db.py` 中添加 `user_themes` 表
2. 创建 `core/storage/theme_repo.py`（继承 `BaseRepository`）
3. 实现 CRUD：`get_all()`, `get_by_id()`, `save()`, `delete()`, `set_active()`
4. 在 `ThemeManager` 中添加 `load_user_theme(id)` 方法

**验证**：单元测试 CRUD 操作

---

### Phase 3: 主题编辑器 UI（2-3 天）

**目标**：用户可编辑颜色并实时预览

**任务**：
1. 新建 `ui/theme_editor.py` - `ThemeEditorDialog`
2. 简单模式（默认）：
   - 主色、背景色、卡片色、文字色、按钮色、边框色、成功色、警告色、错误色
   - 每个颜色旁显示影响范围说明
3. 高级模式（可折叠）：
   - 所有 Token 分组展示
   - Primary/Secondary/Accent 色阶
   - 文字层级、边框、语义色
4. 实时预览区：
   - 模拟聊天面板、按钮、输入框、卡片
   - 修改颜色即时更新预览
5. 对比度校验：
   - 显示 text/background, on_primary/primary 等对比度
   - 绿色/黄色/红色指示可读性
6. 保存/取消/恢复按钮

**验证**：手动测试编辑→预览→保存→应用流程

---

### Phase 4: 宠物 UI 颜色提取（2 天）

**目标**：上传宠物图片后自动生成主题草稿

**任务**：
1. 新建 `core/services/theme_generator.py` - `ThemeGenerator`
2. 实现颜色提取：
   - 方案 A：使用 `colorthief-py` 库（简单，依赖少）
   - 方案 B：自实现 K-Means（更灵活，无依赖）
3. 语义映射逻辑：
   ```
   提取 5-8 个主色
   ↓
   按明度/饱和度分类
   ↓
   映射到 Token：primary, background, text, accent, surface
   ↓
   生成色阶（hover/pressed/soft）
   ↓
   做可读性修正
   ↓
   输出完整 ThemeColors
   ```
4. 在 `PetSettingsPanel` 中添加"从当前宠物生成主题"按钮
5. 生成后跳转到 `ThemeEditorDialog` 预览/编辑

**验证**：测试多张图片提取效果

---

### Phase 5: 集成与打磨（1-2 天）

**目标**：打通完整流程

**任务**：
1. 在 `SettingsWindow` 的"主题设置"Tab 中：
   - 显示预制主题列表
   - 显示"我的主题"列表
   - 添加"从宠物 UI 生成"入口
2. 用户主题操作：
   - 启用、重命名、编辑、复制、删除
3. 预制主题操作：
   - 启用、复制为我的主题后编辑
4. 全局应用逻辑：
   - 选择主题 → 读取 Token → `ThemeManager.apply_theme()` → 信号传播
5. 主题来源标记：
   - "由宠物 UI 生成" / "手动创建" / "从预制复制"

**验证**：端到端测试完整流程

## 4. 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 图片提取颜色质量差 | 用户体验差 | 做语义映射+可读性修正，提供编辑器让用户调整 |
| 对比度校验过严 | 用户被限制 | 只提示不阻止，提供"一键修复"建议 |
| 编辑器预览与实际不一致 | 用户困惑 | 预览区使用真实 QSS 生成函数 |
| 用户主题过多导致 DB 膨胀 | 性能问题 | 限制最多 20 个用户主题，软删除 |
| 旧主题数据迁移 | 兼容性 | 新增字段设默认值，不破坏现有数据 |

## 5. MVP 范围建议

### ✅ 必须做（Phase 1-3）

- 补全 ThemeColors 缺失 Token
- 用户主题 CRUD 存储
- 主题编辑器（简单模式）
- 对比度校验提示
- 预制主题与用户主题统一管理

### ⚠️ 可选做（Phase 4）

- 宠物 UI 颜色自动生成
- 语义映射算法

### ❌ 暂不做

- AI 命名（"金毛暖阳"等）
- 多风格同时生成
- 深色模式自动生成
- 主题分享/导入
- 色彩锁定
- 局部风格 Token（圆角、阴影等）

## 8. 结论

该方案在现有架构上**高度可行**：

1. `ThemeColors` + `ThemeManager` 已提供坚实的 Token 基础
2. `SettingsRepository` + SQLite 已有存储能力
3. `PetAssetService` 已有图片上传流程
4. 主要工作量在：编辑器 UI、颜色提取算法、集成调试

建议按 Phase 1 → 2 → 3 → 5 顺序推进，Phase 4 可作为增强功能后续加入。
