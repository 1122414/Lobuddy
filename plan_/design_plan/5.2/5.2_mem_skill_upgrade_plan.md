# Lobuddy 5.2 Memory & Skill Upgrade Plan

**Date:** 2026-05-02  
**Target branch:** main  
**Scope:** Memory kernel, conversation memory, user/system memory, skill lifecycle, nanobot integration boundary

## 1. Executive Summary

Lobuddy 现在已经有三套相关能力：

- `core/memory/*`：Lobuddy 自己的用户画像记忆，写入 `data/memory/USER.md`，通过消息数量和强信号触发更新。
- `core/storage/chat_repo.py`：SQLite 保存会话和消息，当前主要用于历史展示，还没有形成可检索召回层。
- `lib/nanobot`：自带 `workspace/USER.md`、`SOUL.md`、`memory/MEMORY.md`、`memory/history.jsonl`、Dream、Consolidator、`workspace/skills/*/SKILL.md`。

当前最大问题不是“没有记忆”，而是“记忆权威来源分裂”。Lobuddy 的用户画像和 nanobot 的用户画像写在不同位置，skill 面板和 nanobot skill 也不是同一套系统。下一步不建议完全抛弃 nanobot，而是把 nanobot 降级为执行引擎和兼容层：Lobuddy 自己掌握长期记忆、检索、合并、删除、过期、skill 生命周期；nanobot 继续提供 agent loop、工具调用、`SKILL.md` 格式、progressive disclosure 思路。

## 2. Design Decision

### 2.1 Keep

- 保留 nanobot 作为 AI 执行边界，继续通过 `core/agent/nanobot_adapter.py` 接入。
- 保留 nanobot `SKILL.md` 目录格式，兼容 `workspace/skills/{skill}/SKILL.md`。
- 借鉴 nanobot Dream 的两阶段整理思路：先分析，再做受控修改。
- 借鉴 Hermes Agent 的 skill 机制：成功案例可以转为 skill，但 skill 必须能更新、删除、过期和审计。

### 2.2 Replace Or Own In Lobuddy

- Lobuddy 自己维护权威记忆库，SQLite 为主，Markdown 只是可读投影。
- Lobuddy 自己决定每次 prompt 注入什么记忆，不把上下文组装完全交给 nanobot。
- Lobuddy 自己维护 skill 状态、版本、使用统计、过期策略、启用/禁用。
- Lobuddy 自己实现自动 skill candidate 生成、评估、合并、删除。

### 2.3 Do Not Do

- 不让 Lobuddy 和 nanobot 各自独立写 `USER.md`。
- 不把所有历史都塞进 prompt。
- 不让 skill 只增不减。
- 不让自动总结直接覆盖用户记忆和 skill 文件。
- 不把 UI 展示型 `core/skills/skill_registry.py` 当作真正的 agent skill 系统继续扩大。

## 3. Target Architecture

```text
User Message
    |
    v
ChatRepository(SQLite raw conversation)
    |
    +--> MemoryExtractor(strong signal / periodic / session end)
    |        |
    |        v
    |    memory_item table
    |
    +--> ConversationSummarizer(rolling + session summary)
    |        |
    |        v
    |    conversation_summary table
    |
    +--> SkillCandidateExtractor(successful cases / repeated patterns)
             |
             v
        skill_candidate table
             |
             v
        SkillManager(create / patch / disable / delete)

Before nanobot run:
    MemorySelector + SkillSelector
        |
        v
    PromptContextBundle
        |
        v
    NanobotAdapter.run_task()
```

## 4. Memory Model

### 4.1 Memory Types

| Type | Purpose | Storage | Prompt Strategy |
|---|---|---|---|
| `user_profile` | 用户偏好、沟通风格、边界、长期目标 | SQLite + `data/memory/USER.md` projection | 高频小体积注入 |
| `system_profile` | Lobuddy 的行为约定、用户对桌宠的设定 | SQLite + `data/memory/SYSTEM.md` projection | 稳定注入 |
| `project_memory` | 当前 repo、工具链、踩坑记录、成功修复路径 | SQLite + `data/memory/PROJECT.md` projection | 按项目/关键词召回 |
| `conversation_summary` | 会话滚动摘要、阶段结论 | SQLite | 当前 session 注入 |
| `episodic_memory` | 过去对话中的具体事件和决策 | SQLite FTS/vector-ready | 检索召回，不常驻 |
| `procedural_memory` | 可复用操作流程 | skill 文件 + registry | 通过 skill selector 激活 |

## 5. Memory Update Pipeline

### 5.1 Triggers

- Strong signal: “记住这个”、“以后都这样”、“我不喜欢”、“不要再”、“默认使用”等。
- Every N user messages: 默认 6 条，可配置。
- Session end: 结束会话时整理。
- Task success: 工具链复杂且成功时提取项目/流程记忆。
- Task failure then recovery: 失败后找到正确方案时优先提取 procedural candidate。
- Daily consolidation: 每天低频整理冲突、过期、合并。

### 5.2 Extract

新增 `core/memory/memory_extractor.py`：

- 输入最近消息、当前 profile、任务结果、工具使用记录。
- 输出结构化 patch，不直接写文件。
- Patch action 支持 `add`、`update`、`merge`、`remove`、`deprecate`、`uncertain`。
- 每条 patch 必须包含 `memory_type`、`content`、`confidence`、`reason`。

### 5.3 Validate

新增 `core/memory/memory_validator.py`：

- 拒绝 API key、token、邮箱等敏感内容。
- 拒绝低置信度长期记忆。
- 拒绝“今天聊了什么”这种无长期价值内容。
- 对冲突信息标记 `needs_review`，不直接覆盖。
- 对用户明确纠正的信息允许替换旧项。

### 5.4 Persist

新增 `core/memory/memory_repository.py`：

- SQLite 是权威存储。
- Markdown projection 只用于人类查看和必要时给 nanobot 兼容。
- 每次更新写 `memory_event` 或复用 app 日志，方便回滚和审计。

### 5.5 Select For Prompt

新增 `core/memory/memory_selector.py`：

- 固定注入：少量 `user_profile`、`system_profile`。
- 检索注入：根据当前用户消息召回 `project_memory`、`episodic_memory`。
- 会话注入：当前 session 的 rolling summary。
- 总预算建议：初期控制在 2k-4k chars，不超过 prompt 的 20%。

## 6. Skill Lifecycle

### 6.1 Skill States

| Status | Meaning |
|---|---|
| `draft` | 自动生成或人工创建，但未启用 |
| `active` | 可被 selector 注入/推荐 |
| `needs_review` | 有冲突、失败率高、过期或内容膨胀 |
| `disabled` | 暂停使用，文件保留 |
| `archived` | 不再使用，文件移入 archive |
| `deleted` | 从 registry 删除，文件可按策略删除 |

### 6.2 Creation Criteria

自动 skill candidate 只在满足以下条件之一时生成：

- 同类任务重复出现至少 2 次，并且有稳定成功路径。
- 单次任务复杂度高，工具调用多，且最终成功。
- 用户明确说“以后遇到这种情况就这样做”。
- 出现非显然解决方案，比如失败后换了工具、参数、路径才成功。
- 某类文件/API/项目流程需要固定步骤或脚本。

不应该创建 skill 的情况：

- 只是用户个人偏好，放入 user memory。
- 只是当前项目事实，放入 project memory。
- 能从源码、README、git history 直接得到。
- 一次性任务，无复用价值。
- 内容过长但没有明确触发条件。

### 6.3 Skill Format

继续使用 nanobot/Codex 兼容的 `SKILL.md`：

```markdown
---
name: example-skill
description: Concise description. Use when ...
---

# Example Skill

## Workflow

1. Do the smallest reliable discovery step.
2. Use the existing project pattern.
3. Validate with the narrowest relevant command.

## References

- Read `references/schema.md` only when database details are needed.
```

原则：

- frontmatter 只保留 `name`、`description`，除非确实需要 `always` 或 metadata。
- `description` 必须写清楚触发场景。
- `SKILL.md` 保持短，详细内容放 `references/`。
- 重复脚本放 `scripts/`，并必须有 smoke test。
- 不生成 `README.md`、`CHANGELOG.md`、`QUICK_REFERENCE.md` 等杂文件。

### 6.4 Auto Update And Deletion

新增 `core/skills/skill_manager.py`：

- `create_candidate()`
- `approve_candidate()`
- `create_skill()`
- `patch_skill()`
- `disable_skill()`
- `archive_skill()`
- `delete_skill()`
- `record_skill_result()`
- `review_stale_skills()`

定期维护规则：

- 30 天未使用：降低排序权重。
- 60 天未使用：标记 `needs_review`。
- 90 天未使用且没有用户 pin：自动 disable。
- 失败率超过 40% 且使用次数 >= 5：标记 `needs_review`。
- 新 skill 与旧 skill 触发描述高度相似：进入 merge review。
- skill 文件超过 500 行：要求拆 references 或压缩。
- 依赖失效、命令不存在、路径不存在：disable 并记录事件。

## 7. Nanobot Integration Plan

### 7.1 Short Term

修改 `core/agent/nanobot_adapter.py`：

- 不再直接拼 `UserProfileService.get_profile_context()` 到 prompt 前面。
- 改为 `MemoryContextService.build_context_bundle(user_message, session_id)`。
- bundle 包含：
  - `profile_context`
  - `session_summary`
  - `retrieved_memories`
  - `active_skill_summary`

示例注入：

```text
## Lobuddy Memory Context

### User Profile
...

### Current Session Summary
...

### Relevant Past Memory
...

### Available Skills
...
```

### 7.2 Compatibility Projection

保留或生成以下文件供 nanobot 兼容读取：

- `workspace/USER.md`：由 Lobuddy 从 SQLite projection 出来。
- `workspace/SOUL.md`：由 Lobuddy 控制，用户可编辑或 UI 配置。
- `workspace/memory/MEMORY.md`：由 Lobuddy projection 项目/系统记忆。
- `workspace/skills/*/SKILL.md`：由 Lobuddy SkillManager 管理。

这些文件不是权威数据源，不能让 nanobot Dream 随意覆盖。若需要使用 nanobot Dream，必须改为“产出 patch proposal”，再交给 Lobuddy validator。

## 9. Prompt Templates

### 9.1 Memory Extraction Prompt

```text
Analyze recent conversation and propose memory patches.

Only extract durable information that prevents the user from repeating themselves.
Prefer corrections, preferences, boundaries, ongoing projects, decisions, and non-obvious successful solutions.
Skip one-off chatter, secrets, raw logs, and facts already present.

Return JSON array only:
[
  {
    "memory_type": "user_profile|system_profile|project_memory|episodic_memory|procedural_memory",
    "action": "add|update|merge|remove|deprecate|uncertain",
    "content": "...",
    "confidence": 0.0,
    "importance": 0.0,
    "reason": "..."
  }
]
```

### 9.2 Skill Candidate Prompt

```text
Decide whether this successful task should become a reusable skill.

Create a candidate only if the workflow is reusable, non-obvious, and likely to help future tasks.
Do not create a skill for simple facts, user preferences, project facts, or one-off work.

Return JSON only:
{
  "should_create": true,
  "proposed_name": "short-hyphen-name",
  "title": "...",
  "rationale": "...",
  "description": "What it does. Use when ...",
  "skill_md": "..."
}
```

## 10. Migration Strategy

1. Read existing `data/memory/USER.md`.
2. Parse sections into `memory_item(memory_type='user_profile')`.
3. Read `workspace/USER.md` if present.
4. Merge exact duplicates.
5. Conflicting entries become `needs_review`.
6. Write fresh projections.
7. Keep old files as `.bak` for one release.

## 11. Risks

| Risk | Mitigation |
|---|---|
| 自动记忆写入错误事实 | confidence threshold、source tracking、review state |
| prompt 越来越大 | selector budget、FTS recall、stale cleanup |
| skill 越积越多 | TTL、usage scoring、review、disable/delete |
| nanobot Dream 覆盖 Lobuddy 记忆 | projection-only policy，Dream 只能产出 proposal |
| 敏感信息被记住 | sanitizer + validator + tests |
| UI 和 agent skill 再次分裂 | `SkillManager` 做唯一数据源 |

## 12. Suggested First PR

第一步 PR 不要做自动 skill 创建，先打地基：

- 新增 `memory_item` 表和 repository。
- 将现有 `UserProfileService` 迁移到 `MemoryService` facade。
- 实现 `USER.md` projection。
- `NanobotAdapter` 改为使用 `MemoryService.build_prompt_context()`。
- 保持原有 UI 行为不变。

## 13. Final Recommendation

Lobuddy 应该采用“自有记忆内核 + nanobot 兼容执行层”的路线。nanobot 的 Dream、MemoryStore、SkillsLoader 方向值得借鉴，但它不是 Lobuddy 的产品级权威记忆系统。下一阶段的重点不是让模型记更多，而是让记忆有类型、有来源、有置信度、有生命周期，并且让 skill 能创建、更新、合并、禁用、删除。
