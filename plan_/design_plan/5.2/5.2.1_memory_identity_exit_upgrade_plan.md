# Lobuddy 5.2.1 记忆系统升级计划 —— 身份记忆、退出分析与机制优化

**Date:** 2026-05-02
**Target:** `main`
**Scope:** 身份记忆即时持久化、退出时AI记忆分析、记忆机制深层优化
**Status:** Design Phase — 待评审后实施

---

## 1. 问题诊断

### 1.1 核心问题：身份记忆丢失

用户场景：
- 用户说"我是小明，你是哈基宝"
- AI 本轮能正确称呼，但重启后完全忘记
- AI 声称使用"Dream机制"，但 Lobuddy 并未启用 Dream

根因分析：
1. **触发条件过严**：`_sync_strong_signal_memory` 只在 `should_update_profile` 返回 True 时执行
2. **Strong Signal 列表缺失**：`user_profile_triggers.py` 的 `STRONG_SIGNALS` 包含"我叫"但不包含"我是"、"你是"
3. **AI 幻觉**：nanobot 底层有 Dream 代码，AI 通过 `read_file` 工具读取后误认为自己有 Dream 功能
4. **异步链路不可靠**：profile update 是异步 AI 调用，用户可能在完成前已退出

### 1.2 退出时记忆丢失

当前 `on_exit_requested()` 只保存：
- 宠物位置 (`position_x`, `position_y`)
- 窗口大小 (`task_panel_width`, `task_panel_height`)

**完全不涉及**：
- 会话总结
- 用户偏好分析
- 身份确认
- 记忆同步

### 1.3 记忆机制缺陷

- **去重逻辑弱**：子字符串匹配无法处理语义等价（如"User's name is Bob" vs "Name: Bob"）
- **无冲突解决**：同一话题的多条记忆可能矛盾，没有标记 review 的机制
- **无过期策略**：记忆一旦写入就永久 active，没有生命周期管理
- **注入无优先级**：所有记忆平级注入，重要信息可能被稀释

---

## 2. 设计目标

| 目标 | 指标 |
|------|------|
| 身份记忆即时持久化 | 用户说"我是XX"后，重启必须记得 |
| 退出时记忆不丢 | exit 时自动分析并保存本次会话要点 |
| AI 不幻觉 Dream | prompt 中明确告知不使用 Dream |
| 记忆质量可管理 | 有去重、冲突、过期机制 |

---

## 3. 方案设计

### 3.1 身份记忆：方案 A + C

#### 方案 A：即时正则提取（覆盖 90% 场景）

**时机**：`run_task` 收到用户消息后、发送给 AI 之前

**规则**：不依赖 strong signal，只要消息中包含身份信息就直接提取

```
用户身份触发词：
- "我叫XX" / "我是XX" / "我的名字是XX"
- "以后叫我XX" / "叫我XX就好"
- "你可以叫我XX"

宠物身份触发词：
- "你叫XX" / "你是XX" / "你的名字是XX"
- "以后叫你XX"
```

**行为**：
1. 正则提取名字
2. 同步写入 SQLite `memory_item`
3. 标记 `source="identity_extract"`
4. 继续正常对话流程（零延迟）

**边界情况**：
- 如果名字已存在且相同：跳过
- 如果名字已存在但不同：更新（旧记录 deprecate，新记录 add）
- 如果提取为空或纯数字/符号：丢弃

#### 方案 C：AI 兜底分析（覆盖复杂表达）

**时机**：对话结束时（exit 或 session 切换时）

**触发条件**：
- 本次会话消息数 > 3
- 或用户发送了包含"记住"、"以后"等词的消息

**行为**：
1. 调用轻量级 AI 分析本次会话
2. 识别隐含身份信息（如"从今往后你就叫我大哥"）
3. 返回结构化 patch
4. 经过 validator 后写入 SQLite

**与方案 A 的关系**：
- 方案 A 先写一条 immediate 记录
- 方案 C 在退出时可能补充或修正
- 如果方案 C 发现新信息与 A 冲突，用 confidence 高的覆盖

#### Prompt 中明确禁用 Dream

在 `build_injection_text` 中添加声明：

```
## Lobuddy Memory Context

以下是我的实时记忆。我不使用 Dream 机制，所有记忆都是即时持久化的。
如果用户询问记忆机制，请如实回答：
- 我没有 Dream 功能
- 我的记忆来自 SQLite 数据库
- 所有记忆都是即时写入的

---
```

### 3.2 退出时 AI 记忆分析

#### 时机选择

**策略：延迟退出 + 后台分析**

用户点击 exit 后：
1. UI 立即响应（窗口关闭、托盘隐藏）
2. 后台启动一个 3 秒内的异步分析任务
3. 分析完成后写入 SQLite
4. 然后真正退出进程

**为什么不是同步等待**：
- AI 分析可能需要 2-5 秒，同步等待会让退出感觉卡顿
- 用户已经看到窗口关闭，心理预期是"已退出"
- 后台分析即使失败，也不影响用户体验

#### 分析内容

**必须提取**：
- 用户身份（名字、称呼偏好）
- 宠物身份（名字、性格设定）
- 用户明确表达的偏好（"我喜欢蓝色"）

**可选提取**（如果 token 预算允许）：
- 项目相关事实（"我们在做XX项目"）
- 沟通风格（"用户喜欢简洁回答"）
- 边界（"用户不喜欢被叫'亲'"）

#### 实现流程

```
on_exit_requested()
  ├─ 保存外观/位置（现有逻辑）
  ├─ 关闭 UI（现有逻辑）
  ├─ 启动 exit_analyzer 后台任务
  │   ├─ 读取本次会话消息
  │   ├─ 调用轻量级 AI 分析
  │   ├─ 验证并过滤结果
  │   └─ 写入 SQLite
  └─ 3秒后强制退出（现有逻辑）
```

**失败处理**：
- 如果分析任务 3 秒内未完成：丢弃，不影响退出
- 如果 AI 返回无效 JSON：丢弃，记录警告日志
- 如果 SQLite 写入失败：不阻塞退出

### 3.3 记忆机制优化

#### 优化 1：语义去重

**问题**：子字符串匹配无法处理语义等价

**方案**：引入 embedding-based 相似度检测

```python
def _find_semantically_similar(content: str, threshold: float = 0.85):
    """使用 embedding 找到语义相似的记忆。"""
    # 短期方案：调用 LLM API 计算 embedding（成本高）
    # 长期方案：本地轻量模型（如 sentence-transformers/all-MiniLM）
```

**权衡**：
- embedding 计算有成本，不适合每条消息都做
- 只在方案 C（退出分析）中使用
- 方案 A（即时提取）仍用子字符串匹配（速度快）

#### 优化 2：冲突检测与标记

**规则**：
- 同一 `memory_type` + `scope` 下，如果两条 active 记忆内容矛盾
- 新记忆 confidence > 旧记忆 confidence：覆盖旧记忆
- 新记忆 confidence <= 旧记忆 confidence：标记为 `needs_review`

**示例**：
```
旧记忆："用户喜欢猫" (confidence=0.8)
新记忆："用户讨厌猫" (confidence=0.9)
→ 旧记忆 deprecate，新记忆 active

旧记忆："用户喜欢猫" (confidence=0.8)
新记忆："用户讨厌猫" (confidence=0.7)
→ 新记忆标记 needs_review，人工确认前不注入 prompt
```

#### 优化 3：记忆生命周期

引入 `expires_at` 字段（已有但未使用）：

| 记忆类型 | 默认有效期 | 续期条件 |
|---------|-----------|---------|
| 身份信息 | 永久 | 用户确认后续期 |
| 偏好 | 90天 | 再次被提及时续期 |
| 项目事实 | 项目结束时 | 用户明确说"这个项目结束了" |
| 临时事件 | 30天 | 不续期 |

**清理策略**：
- 每日 maintenance 检查过期记忆
- 过期记忆标记为 `DEPRECATED`，不删除（保留审计）
- 用户可在设置面板查看和恢复过期记忆

#### 优化 4：注入优先级

当前 `MemorySelector` 平级注入所有记忆。改进：

```python
class MemoryItem:
    # 新增字段
    priority: int = 50  # 1-100，越高越先注入
    
# 身份记忆：priority=90
# 偏好：priority=70
# 项目事实：priority=60
# 历史事件：priority=40
```

在预算不足时，优先保留高 priority 记忆。

#### 优化 5：记忆溯源与可解释性

每条记忆记录：
- `source`: 来源（bootstrap / identity_extract / ai_analysis / manual）
- `source_session_id`: 来自哪次会话
- `source_message_id`: 来自哪条消息

UI 展示：
- 用户点击"我的记忆"面板时，显示每条记忆的来源和创建时间
- 用户可质疑某条记忆（"我怎么可能喜欢猫？"）
- 质疑后该记忆标记 `needs_review`，不再注入

## 6. 验收标准

### 6.1 身份记忆

| 场景 | 预期结果 |
|------|---------|
| 用户说"我是小明，你是哈基宝" | 重启后 AI 知道"你是小明，我是哈基宝" |
| 用户说"叫我大哥就行" | 重启后 AI 称呼用户"大哥" |
| 用户说"以后别叫我小明了，叫我大明" | 重启后 AI 称呼"大明"，旧记忆已更新 |
| 用户问"你用什么记忆机制" | AI 回答"SQLite 即时记忆，没有 Dream" |

### 6.2 退出分析

| 场景 | 预期结果 |
|------|---------|
| 聊天 5 分钟后 exit | 3 秒内退出，无卡顿，记忆已保存 |
| 聊天 1 条消息后 exit | 不触发分析（< min_messages），直接退出 |
| AI 分析失败 | 不影响退出，记录 warning 日志 |

### 6.3 机制优化

| 场景 | 预期结果 |
|------|---------|
| 记忆冲突 | 高 confidence 覆盖低 confidence |
| 记忆过期 | 过期后不再注入 prompt，但可在面板查看 |
| 预算不足 | 优先保留身份信息 |

## 7. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 正则提取误匹配（如"我是程序员"提取为名字"程序员"） | 1. 过滤常见词（程序员、学生、老师等）2. AI 兜底分析时修正 |
| 退出分析导致退出延迟 | 3 秒超时强制退出，分析任务可丢弃 |
| AI 仍然幻觉 Dream | prompt 中明确声明 + 记忆注入格式声明 |
| 记忆冲突误判 | 先实现简单规则，后续引入 embedding |

## 8. 附录：Prompt 模板

### 8.1 退出分析 Prompt

```text
分析以下对话，提取用户和宠物的身份信息以及用户明确表达的偏好。

**只提取以下内容**：
1. 用户的名字或称呼（如"小明"、"大哥"）
2. 宠物的名字（如"哈基宝"）
3. 用户明确说的偏好（如"我喜欢蓝色"、"别叫我亲"）

**忽略**：
- 临时性话题
- 工作任务细节
- 情绪表达

**输出格式**：JSON
{
  "identities": [
    {"type": "user_name", "value": "...", "confidence": 0.9},
    {"type": "pet_name", "value": "...", "confidence": 0.9}
  ],
  "preferences": [
    {"content": "...", "confidence": 0.8}
  ]
}

**对话**：
{conversation}
```
