**Date:** 2026-05-02  
**Analyst:** Sisyphus (Code Agent)  
**Target:** `plan_/design_plan/5.2/5.2_mem_skill_upgrade_plan_start.md`  
**Scope:** Assess plan feasibility against current codebase state

---

## 1. Executive Summary

The 5.2 plan proposes a foundational shift: **move authoritative memory storage from markdown files into SQLite**, introduce a typed memory model with lifecycle management, and build a real skill registry with CRUD + auto-candidate generation. After exhaustive codebase inspection, the plan is **technically feasible and architecturally sound**, but carries **medium implementation risk** due to:

1. **No existing migration system** — new tables must be added manually.
2. **Nanobot internal API coupling** — `NanobotGateway` relies on nanobot private attributes.
3. **FTS5 availability uncertainty** — depends on Python's sqlite3 compilation flags.
4. **Prompt budget control complexity** — easy to get wrong; needs rigorous testing.

**Recommendation:** Proceed with **Milestone 1 only** in the first PR. Defer M2–M5 until M1 stabilizes and the new memory boundary is proven in production.

## 3. Feasibility by Milestone

### 3.1 M1: Unify Memory Boundary — ✅ HIGHLY FEASIBLE

**Files to create:**
- `core/memory/memory_schema.py` — Pydantic models for `MemoryItem`, `ConversationSummary`.
- `core/memory/memory_repository.py` — SQLite CRUD for `memory_item` table.
- `core/memory/memory_service.py` — Facade over repository + projection.
- `core/memory/memory_projection.py` — Markdown projection to `data/memory/*.md` and `workspace/*.md`.
- `tests/test_memory_repository.py`
- `tests/test_memory_service.py`

**Files to modify:**
- `core/storage/db.py` — Add `memory_item` table creation in `init_database()`.
- `core/agent/nanobot_adapter.py` — Replace `profile_ctx + prompt` with `MemoryContextService.build_context_bundle()`.
- `core/memory/user_profile_service.py` — Wrap as compatibility facade.

**Technical assessment:**
- **Table schema:** Plan's `memory_item` schema is straightforward. Aligns with existing conventions (TEXT PRIMARY KEY, ISO datetime strings, REAL for scores).
- **Migration path:** Add table creation to `Database.init_database()`. Use `_ensure_column` for future column additions. Acceptable given no Alembic.
- **Projection layer:** Simple — render SQLite rows into markdown sections. Can reuse `_render_markdown` logic from `UserProfileManager`.
- **NanobotAdapter change:** Low risk. Replace 3 lines of prefix concatenation with a service call. Adapter already has `set_profile_service()` hook.
- **Backward compatibility:** `UserProfileService` can delegate to new `MemoryService` while keeping same public API. UI behavior unchanged.

**Estimated effort:** 2–3 days (including tests).

**Open questions:**
1. Should `memory_item` use `rowid` for FTS5 or an explicit `id TEXT PRIMARY KEY`?  
   → **Recommendation:** Use `id TEXT PRIMARY KEY` for consistency with existing tables, but create FTS5 with `content_rowid='rowid'` (FTS5 requires integer rowid). This means `memory_item` should use `INTEGER PRIMARY KEY` or accept that `rowid` is separate from `id`. **This needs a decision before implementation.**

2. Should existing `chat_session` / `chat_message` be linked to `memory_item.source_session_id` via foreign key?  
   → **Recommendation:** No FK constraint. Use `TEXT` for flexibility (session IDs may come from nanobot too). Add index instead.

### 3.2 M2: Conversation Summary And Retrieval — ✅ FEASIBLE WITH CAVEATS

**Files to create:**
- `core/memory/conversation_summarizer.py`
- `core/memory/memory_selector.py`
- `tests/test_conversation_summary.py`
- `tests/test_memory_selector.py`

**Files to modify:**
- `core/storage/chat_repo.py` — Potentially add summary-related queries.

**Technical assessment:**
- **Rolling summary:** Can hook into `HistoryCompressor` or run independently. Since `HistoryCompressor` already operates on nanobot sessions, **keep them separate** — let `ConversationSummarizer` read from `chat_repo` and write to `conversation_summary` table.
- **FTS5 recall:** SQLite FTS5 is available in Python 3.11+ on most platforms (Windows CPython includes it). **BUT** some minimal builds (Alpine Linux, custom compiles) may exclude it. **Mitigation:** Feature-detect at startup; if FTS5 unavailable, fall back to `LIKE` search with degraded performance.
- **Prompt budget control:** Critical. The plan suggests 2k–4k chars, ≤20% of prompt. Needs a `TokenBudget` utility class. This is **easy to implement, hard to tune** — requires production observation.

**Risk:**
- FTS5 availability on target platforms. **Mitigation:** Add `Database.has_fts5()` check at init; disable FTS5 features if unavailable.
- Summary quality depends on LLM. Bad summaries = worse retrieval. **Mitigation:** Keep original messages as fallback; summary is additive, not replacement.

**Estimated effort:** 2–3 days (including tests).

### 3.4 M4: Skill Candidate Automation — ⚠️ MODERATE RISK

**Files to create:**
- `core/skills/skill_candidate_extractor.py`
- `core/skills/skill_validator.py`
- `core/skills/skill_maintenance.py`
- `tests/test_skill_candidate_extractor.py`
- `tests/test_skill_maintenance.py`

**Technical assessment:**
- **Candidate extraction:** Requires analyzing task results (success/failure, tool usage, complexity). `AgentResult` already has `tools_used`, `success`, `raw_output`. Can hook into `TaskManager` completion.
- **Duplicate detection:** Need similarity metric for skill descriptions. Simple approach: embedding-based cosine similarity (requires vector storage or API call). **Simpler approach:** Exact match on proposed name + description fuzzy match. **Recommendation:** Start simple — exact name match + keyword overlap. Upgrade later.
- **Auto-creation policy:** Plan suggests default `draft` status, configurable auto-approve. This is safe.
- **Sensitive info in skills:** Need sanitizer (reuse `_sanitize_line` from `user_profile_manager.py`).

**Risk:**
- **False positives:** May generate candidates for non-reusable tasks. **Mitigation:** Strict criteria (plan lists 5 creation conditions + 5 exclusion conditions). Start conservative.
- **LLM cost:** Candidate extraction requires an extra LLM call per task. **Mitigation:** Only run for tasks with `len(tools_used) > 2` or user strong signal.

**Estimated effort:** 4–5 days (including tests and tuning).

## 4. Cross-Cutting Concerns

### 4.1 Database Migration Strategy

Since Lobuddy has **no migration framework**, adding tables must be done carefully:

```python
# In Database.init_database(), after existing tables:
cursor.execute("""
    CREATE TABLE IF NOT EXISTS memory_item (
        id TEXT PRIMARY KEY,
        ...
    )
""")
# FTS5 virtual table (conditional)
if self._has_fts5():
    cursor.execute("""
        CREATE VIRTUAL TABLE IF NOT EXISTS memory_item_fts USING fts5(...)
    """)
```

**Migration from existing USER.md:**
1. On first run with new code, read `data/memory/USER.md`.
2. Parse sections into `memory_item(memory_type='user_profile')` rows.
3. Write SQLite.
4. Rename `USER.md` → `USER.md.bak`.
5. Generate fresh projection from SQLite.

**This is safe and reversible.**

### 4.2 FTS5 Availability

```python
def _has_fts5(self) -> bool:
    try:
        with self.get_connection() as conn:
            conn.execute("CREATE VIRTUAL TABLE _fts5_test USING fts5(x)")
            conn.execute("DROP TABLE _fts5_test")
            return True
    except sqlite3.OperationalError:
        return False
```

**Fallback:** If FTS5 unavailable, use `LIKE '%keyword%'` with index on `memory_item(title, content)`. Performance acceptable for <10k rows.

### 4.3 Nanobot Version Lock

The plan's success depends on nanobot's `SkillsLoader` and session APIs remaining stable. Current observations:
- `NanobotGateway` accesses `_loop` (private).
- `SkillsLoader` is public API in nanobot.
- `workspace/` structure is nanobot convention.

**Mitigation:** Pin nanobot submodule to a tested commit. Monitor nanobot changelog for breaking changes in `skills.py` and `session/` modules.

### 4.4 Prompt Budget Enforcement

The most fragile part of the plan. Recommendation for implementation:

```python
class PromptBudget:
    def __init__(self, max_chars: int = 4000, max_percent: float = 0.20):
        self.max_chars = max_chars
        self.max_percent = max_percent

    def allocate(self, prompt: str, bundles: list[MemoryBundle]) -> list[MemoryBundle]:
        budget = min(self.max_chars, int(len(prompt) * self.max_percent))
        selected = []
        used = 0
        for bundle in sorted(bundles, key=lambda b: b.priority, reverse=True):
            if used + len(bundle.content) <= budget:
                selected.append(bundle)
                used += len(bundle.content)
            else:
                break
        return selected
```

**Must be tested with real prompts** to ensure budget is not exceeded.

## 5. Risk Matrix

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| FTS5 unavailable on user's SQLite build | Medium | Low | Feature-detect; fallback to LIKE search |
| nanobot upgrades break `NanobotGateway` | High | Medium | Pin submodule; integration tests |
| Auto-generated memory contains false facts | High | Medium | Confidence threshold + review state + source tracking |
| Prompt budget exceeded, degrading LLM output | Medium | High | Hard budget enforcement + tests; monitor token usage |
| Skill candidate false positives | Medium | Medium | Strict creation criteria; default `draft` status |
| Migration corrupts existing USER.md | Medium | Low | Backup to `.bak`; reversible migration |
| Circular dependency: MemoryService → NanobotAdapter → MemoryService | Medium | Low | Clear layering: `core/memory/` must not import `core/agent/` |
